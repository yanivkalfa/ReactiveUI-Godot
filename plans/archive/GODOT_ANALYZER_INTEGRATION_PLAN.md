# GODOT_ANALYZER_INTEGRATION_PLAN — deep `.guitkx` intelligence via a native gdext binding

Resolves the open backend decision in `GODOT_EDITOR_EXTENSION_PLAN.md §4` (Path 1 vs Path 2).
Status: **ALL PHASES DONE 2026-07-05.** Phase 1 shipped with editor 0.3.0–0.5.0 (markup tier,
pure GDScript). Phases 2–5 shipped with editor **0.6.0** + gdscript-analyzer's
`feat/gdext-binding`: `bindings/gdext` crate (`GdscriptAnalyzer` over `gdscript-session`,
`reloadable=false`, compat floor 4.4), the `release-gdext` per-OS pipeline (win x64 /
linux x64+arm64 / mac universal → `reactive-ui-analyzer-<version>.zip` on each `v*` release),
GDScript ports of virtualDoc/sourceMap + a byte-exact LineIndex (CodeEdit columns confirmed
code-point-based per §9 Q4), and the feature-detected editor bridge with markup-only
degradation. §9 answers as shipped: floor 4.4 (Q1); crate in the analyzer repo (Q2); a separate
optional addon, not bundled (Q3); code-point columns (Q4); parity via the addon suite's
dual-mode e2e + hook-stub/emitter corpus asserts (Q5); unsigned per-arch-lipo'd universal mac
dylib, quarantine step documented (Q6).

Goal: give **both** the native Godot editor addon (`reactive_ui_editor`, pure GDScript) **and** the VS Code /
VS 2022 extensions deep `.guitkx` + embedded-GDScript intelligence, **maximising reuse of what already
exists** and honouring the hard requirements: **no developer-facing TCP server, no per-user config,
in-process / self-contained, and consistency across hosts.**

---

## 1. Decision (one line)

Use a **native GDExtension (godot-rust / gdext) binding of `gdscript-analyzer`** as the embedded-GDScript
backend for the Godot addon, and **port the `.guitkx` markup-LSP logic to GDScript** (reusing the existing
GDScript compiler + host-agnostic JSON). No Node, no TCP, no config; one Rust analysis core shared by all
three hosts.

---

## 2. What already exists (the reuse surface)

**`gdscript-analyzer`** (Rust, `C:/Yanivs/GameDev/gdscript-analyzer`) is a rust-analyzer-style layered
workspace with a **clean binding-agnostic core**:

- `gdscript-base` (POD result types, `FileId`, `TextRange`, `LineIndex`) → `gdscript-syntax` (lexer + lossless
  CST/AST) → `gdscript-db` (salsa VFS, `apply_change`) + `gdscript-api` (engine `extension_api.json` model) →
  `gdscript-hir` (name resolution + type inference + warnings) → **`gdscript-ide`** — *the* public
  `AnalysisHost` / `Analysis` API (engine-neutral, protocol-neutral, POD + byte offsets).
- **`gdscript-session`** — the pure-Rust, wasm-clean crate that both FFI bindings wrap. It owns the
  URI→FileId interner, the `open`/`change`/`close`/`set_project_config` document lifecycle, and
  `serde_json::Value` serialisation (with URI enrichment). **Fully unit-tested with plain `cargo test`.**
- The **two existing bindings are ~200-line delegators with ZERO analysis logic** over `gdscript-session`:
  - napi → `gdscript-ffi` = `@gdscript-analyzer/core` (used by the VS Code / VS 2022 LSP server).
  - wasm → `bindings/wasm` = `@gdscript-analyzer/wasm` (web).

**Capabilities available TODAY** (not diagnostics-only): `diagnostics`, `completions`, `hover`,
`goto_definition`, `find_references`, `rename`, `signature_help`, `inlay_hints`, `document_symbols`,
`workspace_symbols`, `semantic_tokens`, `folding_ranges`, `code_actions`, `format`, `format_range`,
`syntax_tree` — **16 queries.** A gdext binding gets all of them for free.

**The `.guitkx` compiler / lexer / formatter are GDScript**, not Rust: `addons/reactive_ui/guitkx`
(`RUIGuitkx.compile`, `RUIGuitkxLexer`, `RUIGuitkxMarkup`, `RUIGuitkxFormatter`, `RUIGuitkxJsxScan`). The
Phase-1 `reactive_ui_editor` addon already drives compile→diagnostics through them.

**The markup-LSP logic is TypeScript** (`ide-extensions/lsp-server`, ~2,700 LOC of pure offset/string logic:
scanner, context, refs, `workspaceIndex`, `sourceMap`, `virtualDoc`, `semanticTokens`, `events`, plus
`markup.ts`/`formatGuitkx.ts` which are already declared faithful ports of the GDScript
`guitkx_markup.gd`/`guitkx_formatter.gd`). Two data assets are already **host-agnostic JSON**:
`ide-extensions/grammar/guitkx-schema.json` (tag/attr/directive/event vocabulary) and
`ide-extensions/lsp-server/classdb/godot-control.json` (ClassDB dump, itself generated by GDScript).

## 3. The recommendation: native **gdext** binding

A `#[class(base=RefCounted, tool)]` Rust object exposing one `#[func]` per `Session` method, called from
GDScript as e.g. `GdscriptAnalyzer.new().completions(uri, offset)`. It meets all four requirements at once:

| Requirement | How gdext satisfies it |
|---|---|
| No TCP / no server | Loads as a native shared library into the editor process; called by function pointer, in-process. |
| No per-user config | Ships as a prebuilt binary via a `.gdextension` descriptor; nothing to configure. |
| Self-contained / in-process | One object with `GString`→`Dictionary` methods, callable from the existing `@tool` addon. |
| Consistency | Wraps the **same `gdscript-session::Session`** as napi (VS Code/VS) and wasm (web) → byte-identical results. |

Critically this is a **plain data GDExtension, NOT a `ScriptLanguageExtension`**, so it structurally avoids the
segfault / language-registration / hot-reload hazards that plague script-language plugins. Ship with
**`reloadable = false`** (load-once; editor restart only on analyzer upgrade) so gdext's known live-recompile
hot-reload instability never reaches end users.

---

## 4. The shared abstraction (the "one package used by both")

There is nothing new to invent — the shared layer already exists: **`gdscript-session` over `gdscript-ide`.**
All three hosts call the identical `Session` methods and differ **only** in a thin serialisation shim.

**Stable contract**
- Input (push-based, path-free): `open(uri, text, res_path)` / `change(uri, text)` / `close(uri)` /
  `set_project_config(text)`, keyed by URI string.
- Queries: `(uri, byte_offset)` → `serde_json` POD.
- `Diagnostic = { range:{start:u32,end:u32}, severity:'error'|'warning'|'info'|'hint', code:String, message:String, source:'syntax'|'type', fixes:[CodeAction] }`.
- **All offsets are UTF-8 BYTE offsets, half-open `[start,end)`, 0-based lines.** `gdscript-base::LineIndex`
  converts byte ↔ (line,col) ↔ UTF-16; **every client performs this conversion itself.**

**Per-host binding responsibilities** (all ~200-line delegators):
- napi (`gdscript-ffi`): serde→JS free via napi's serde-json feature. *(exists)*
- wasm (`bindings/wasm`): `serde-wasm-bindgen`; must `load_engine_api` at runtime. *(exists)*
- **gdext (new)**: a bespoke `serde_json::Value ↔ Godot Dictionary/Array` shim (there is no built-in
  serde↔Variant bridge); bundles `extension_api.json` via `include_bytes!` (native path, no runtime fetch).

## 5. Markup strategy — the one real fork — **DECIDED: port to GDScript (Option a)**

`.guitkx` *markup* intelligence (tags/attrs/directives/components) is separate from *embedded GDScript*.

**Decision: port the TS markup-LSP logic to GDScript**, reusing the existing GDScript compiler/lexer/formatter
and loading `guitkx-schema.json` + `godot-control.json` directly. **Do NOT** duplicate the `.guitkx` parser in
Rust (Option b would mean a *third* `.guitkx` parser to maintain, in a language with no existing `.guitkx`
compiler).

**Hybrid boundary** (mirrors how the TS server already splits it):
- **Markup** (tags/attrs/directives/components) = **GDScript** (compiler + ported LSP logic + JSON data).
- **Embedded GDScript** (inside `{expr}` / setup / hooks) = the **gdext analyzer**, bridged by a GDScript port
  of `virtualDoc.ts` + `sourceMap.ts` (build a synthetic `.gd` virtual doc, map byte spans, forward to the
  analyzer, remap results back to `.guitkx` coordinates).

**Consistency mechanism:** extend the **already-proven byte-identical golden-fixture corpus** (the discipline
that keeps `formatGuitkx.ts` ↔ `guitkx_formatter.gd` identical, and asserts `markup.ts`/`refs.ts` against
shared fixtures). The GDScript markup-LSP port becomes a **third** implementation validated against the same
corpus (parse trees, semantic-token spans, format output, ref scans).

---

## 6. Why not the alternatives

- **Node subprocess (spawn the TS LSP over stdio/TCP)** — requires shipping/requiring a Node runtime
  (tens of MB, version-sensitive), spawn+supervise+health-check+restart+kill a child process, and communicate
  over stdio or a TCP port — **reintroducing the exact TCP/socket + per-user config this effort exists to
  delete.**
- **wasm-in-Godot (run the wasm build via a wasm-runtime GDExtension)** — the wasm boundary passes only
  int/float, so a `String`→`Dictionary` call needs hand-rolled linear-memory marshalling (what napi/serde
  already solve) plus a heavy third-party runtime dependency. Strictly more moving parts than one native
  `.dll` that speaks `Dictionary`, and only justified if native cross-compile were impossible — it isn't
  (pure Rust, already cross-compiles; salsa built `default-features=false` so the single-threaded core covers
  every desktop platform Godot supports).
- **Godot's built-in GDScript LSP (`:6005`)** — needs a running editor (a TCP dependency), understands only
  plain `.gd` (zero `.guitkx`/embedded knowledge), no cross-host consistency. Fails all four goals.
- **Move markup into Rust (Option b)** — duplicates a compiler the project already ships and maintains in
  GDScript.

## 7. Phased plan (front-loaded — value ships with zero native code)

- **Phase 0 — baseline (DONE).** `reactive_ui_editor`: diagnostics via `RUIGuitkx.compile`, highlighting,
  minimal `<tag` completion. Works with only the GDScript addon installed; no `.gdextension` required.
- **Phase 1 — markup intelligence, pure GDScript, still NO native binary (~2–3 wk).** Port the TS markup-LSP
  pure modules to `addons/reactive_ui_editor/lsp/` (scanner, context, refs→reuse `guitkx_jsx_scan.gd`,
  `workspaceIndex`, `semanticTokens`, `events`, `sourceMap`). Load the two JSON assets directly
  (`FileAccess` + `JSON.parse_string`). Wire richer completion/hover/goto/refs/rename for
  tags/attrs/directives/components into the `CodeEdit` hooks Phase 0 already exposes + the Problems panel.
  Validate every module against the shared golden-fixture corpus.

  **Phase 1 status (2026-07-02) — completion + hover landed & tested.** New `addons/reactive_ui_editor/lsp/`:
  `guitkx_schema.gd` (grammar vocab bundled at `data/guitkx-schema.json`, per-tag props/events/signals from
  **live `ClassDB`** — no `godot-control.json` dump needed since the addon runs in-engine),
  `guitkx_context.gd` (the 5-context classifier: tagName/attrName/attrValue/directive/markup/embedded),
  `guitkx_workspace.gd` (project `.guitkx` component index → file+offset), `guitkx_completion.gd` +
  `guitkx_hover.gd` (UI-free providers). Wired into `guitkx_code_edit.gd`: context-aware completion (host
  tags + user components + per-tag props/events/structural attrs + directives) replacing the Phase-0
  dump-all-Controls stub, reindex-on-save, and `symbol_hovered`→tooltip hover (new `KEY_HOVER` setting,
  guarded for pre-4.4). 39 headless assertions green (`tests/guitkx_lsp_test.gd`) incl. live-ClassDB
  property/event resolution and `<`-as-comparison. **Remaining Phase 1:** go-to-definition + find-references
  + rename wiring (workspace index already maps tag→offset; `refs.ts` scan port pending), semantic-tokens
  overlay, and the shared golden-fixture cross-check (`markup-cases`/`formatter-cases`). Interactive check
  still owed: the hover *tooltip display* is logic-tested but its on-screen rendering needs a live editor.
- **Phase 2 — the gdext crate (~1–1.5 wk).** Add `bindings/gdext` (crate-type `cdylib`, `publish=false`) to
  the analyzer repo, depending on `gdscript-session` + the `godot` crate. `GdscriptAnalyzer #[class(base=RefCounted, tool, init)]`
  holding a `Session`; one `#[func]` per method; the `serde_json::Value ↔ Variant` shim; bundle
  `engine_api.bin` via `include_bytes!`. Unit-test the shim (analysis coverage already guaranteed by
  `gdscript-session`'s tests).
- **Phase 3 — CI + packaging (~1–2 wk).** New `release-gdext` workflow: `cargo build --release` per-OS
  (windows→`.dll`, ubuntu→`.so`, macos→`.dylib` incl. aarch64/universal), `lto=true`, `opt-level=z`,
  `strip=true`. Package a `.gdextension` descriptor (`entry_symbol`, `compatibility_minimum`,
  `reloadable=false`, per-platform `[libraries]`) + matched binaries. Reuse the `release-napi` matrix
  *structure* as a template (not the pipeline).
- **Phase 4 — wire gdext into the addon (~2–3 wk).** Port `virtualDoc.ts` + `sourceMap.ts` to GDScript. In
  the editor, detect embedded context; feature-detect `ClassDB.class_exists("GdscriptAnalyzer")`; if present,
  instantiate it, open/change the virtual doc, forward completions/hover/goto/refs/rename/signature/
  diagnostics at the mapped **byte** offset (convert `CodeEdit` line/col ↔ UTF-8 byte via a ported
  `LineIndex`), remap results back to `.guitkx` coords, merge with the Phase-1 markup diagnostics. **If the
  extension is absent, gracefully degrade to markup-only.** The native binary is an OPTIONAL enhancement.
- **Phase 5 — parity hardening (~1 wk).** Extend the shared corpus to assert GDScript markup outputs match the
  TS ones, and gdext query JSON matches napi query JSON for identical inputs.

## 8. Risks

- **Offset/encoding marshalling** — core speaks UTF-8 byte offsets / 0-based lines; `CodeEdit` is line/column
  (and `get_caret_column` is char/UTF-16-ish). A `LineIndex`-equivalent must be ported to GDScript and used at
  **every** boundary or diagnostics/completions mis-anchor. *High-likelihood, medium-impact; mitigate with
  fixture tests.*
- **gdext hot-reload instability** (documented through 2026) — mitigated by `reloadable=false`; only affects
  the analyzer developer's inner loop.
- **Two markup implementations (GDScript + TS) drift** — mitigated by the shared golden corpus, but real
  recurring maintenance: every grammar change updates GDScript, TS, and fixtures together.
- **gdext ↔ Godot ABI per minor** — `compatibility_minimum` pins a floor; Godot minor releases can break the
  GDExtension ABI; the ClassDB dump regenerates per Godot minor. Needs a per-minor validation step.
- **New per-platform CI matrix** — napi/wasm/cargo-dist pipelines do NOT emit a GDExtension library;
  aarch64/universal-macOS build+sign+notarize adds toolchain friction.
- **The serde→Variant shim is bespoke** — large results (workspace_symbols, semantic_tokens) need care for
  correctness and allocation cost.
- **`Session` holds non-Sync salsa state** — safe only under single-threaded editor use (gdext default). Must
  not call from a threaded context.
