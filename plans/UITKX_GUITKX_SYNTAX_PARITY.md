# .uitkx / .guitkx Syntax & Diagnostics Parity Report

> **Provenance.** 2026-07-03, produced by a six-reader evidence sweep over both codebases (Unity grammar,
> Unity diagnostics, Godot grammar — compiler AND LSP, Godot diagnostics — all four surfaces, docs+samples
> usage, error-message parity vs native Godot) plus a synthesis pass. The four most load-bearing claims
> (exec items 2, 3, 6, and the G5 root) were independently re-verified against source before publishing.
> Bugs found alongside this research are filed in [`BUG_AUDIT.md`](BUG_AUDIT.md) §4 (**G5–G9**) and indexed
> in [`BUG_SPLIT.md`](BUG_SPLIT.md); this report is the *language-level* companion — what should converge,
> what is intentional, and in what order.

**Scope.** `.uitkx` (Unity, ReactiveUIToolKit) and `.guitkx` (Godot, ReactiveUI-Godot) are supposed to be the same markup language modulo C#-vs-GDScript expression syntax. This report synthesizes six evidence readers covering: the Unity grammar and diagnostics as implemented, the Godot grammar and diagnostics as implemented (both the GDScript compiler and the TypeScript LSP), the language as documented/shipped in samples on both sides, and error-message parity against native Godot.

## 1. Executive summary

Worst first:

1. **Godot has two hand-maintained grammar implementations (GDScript compiler vs TypeScript LSP) that provably disagree in at least 8 ways** — missing jsx_scan port, HOST_TAGS alias mismatch, hook-prefix model mismatch, divergent typo-recovery thresholds — while Unity's single shared language-lib makes this class of bug structurally impossible (GJ vs no TS port; GX:19-29 vs SCH:18-51; VD:26-50 vs GX:373/485/632; details Section 5).
2. **The Godot compiler mis-compiles silently in multiple cases**: malformed markup in a branch emits `null  # body parse error` with `ok:true` and zero diagnostics (GX:283-284, 784-785, 1053-1054); the error-severity GUITKX0113 never fails the compile because the gate runs before `_emit` (GX:157-162 vs 896-901); module compiles never run the hard-error gate at all (GX:475-491). Unity fails codegen on every parser error (PIPE:184-207). *(Gate-before-`_emit` re-verified: GX:156-163.)*
3. **Return-statement semantics are opposite and Godot's is unsafe**: Unity takes the *last* top-level `return (...);` (DP:1441-1551); Godot takes the *first* `return (`/`return <` even when it sits inside an `if:` block, indentation-blind, silently dropping everything after it with only a warning (GX:494-526, re-verified) — a conditional early return mis-compiles (overlaps filed bug G6).
4. **Every compiler diagnostic on the Godot side is a plain string with no line/column** (GX:33; GC:31-46), so the LSP sidecar pins all of them to line 0 (SRV:668), the Godot Errors dock shows a path plus a joined array string (PLG:69-79), and the native editor guesses the line by searching for a quoted token (guitkx_diagnostics_renderer.gd:56-72). Unity is line-precise on both surfaces.
5. **Unknown tags/attributes are compile-time-silent on Godot**: lowercase tags emit `V.<tag>` verbatim, PascalCase tags are assumed to be components (GX:703-712, 757-766); the only checks (GUITKX0105/0107) live in the LSP and are gated on a near-miss suggestion existing / ClassDB being loaded (SRV:1568-1596). Unity warns at compile (UITKX0001/0008/0109) and errors in the IDE (0105). Overlaps filed bug G5.
6. **Text interpolation is opposite**: Godot text stops at `{`, so `Count: {n}` interpolates (GM:174-181); Unity text does *not* stop at `{`, so the same markup silently emits the literal string `"Count: {n}"` (MT:340-353, re-verified; CE:2067-2076). Same input, different render, no diagnostic on either side — this one is a Unity-side bug.
7. **Markup comments exist only on Unity** (`//`, `/* */`, `<!-- -->`, UP:463-480) — the Godot parser has none (GM:294) even though Godot's own docs document `#` comments and its doc snippets use `//` inside markup (GDoc Reference:312-315, EventsPage.example.ts:21-148) — those snippets likely do not compile (UNVERIFIED).
8. **The directive-body authoring model is a fork**: Unity control-flow bodies are raw C# requiring `return (<JSX/>);` (UP:157-217, UDoc Reference:230-237); Godot bodies are bare markup (GM:184-288, GDoc Reference:251-258). Documented on both sides so classed intentional, but it is the deepest split for anyone porting UI between engines.
9. **Diagnostic code numbers collide with different meanings** across sides: 0102, 0107, 0110, 0112, 0113, 0114 each mean something different in UITKX vs GUITKX (Section 3), so shared docs/tooling/muscle memory cannot be reused.
10. **Godot silently drops the second and later top-level declarations** in a file, and the LSP formatter *deletes* them plus leading file comments on Format Document (GX:54-69; FMT:49-75, 740-747); Unity errors (UITKX2104/2105, DP:434-456).

## 5. Grammar single-source-of-truth problem

### 5.1 The Godot side has two grammars, and they disagree

The `.guitkx` grammar is implemented **twice by hand**: in GDScript for the compiler (`guitkx.gd`, `guitkx_markup.gd`, `guitkx_jsx_scan.gd`, `guitkx_lexer.gd`) and in TypeScript for the LSP (`markup.ts`, `scanner.ts`, `declScan.ts`, `formatGuitkx.ts`, `virtualDoc.ts`). Where the port was disciplined it works — `guitkx_lexer.gd` and `scanner.ts` are byte-identical *by design* (guitkx_lexer.gd:15-70 = scanner.ts:11-63), and `markup.ts` is line-for-line the markup parser. Everywhere else they have already diverged:

1. **No TS port of `guitkx_jsx_scan.gd` at all** — the LSP virtual doc splices whole expression blocks verbatim (VD:344-355), so nested markup reaches the GDScript analyzer as garbage. Biggest single disagreement; feeds filed **G7/G8**.
2. **HOST_TAGS mismatch** — compiler accepts 9 long-form aliases (`VBoxContainer`, `RichTextLabel`, ..., GX:19-29) missing from the LSP catalog (SCH:18-51): compiles fine, but no highlight/completion/attr-check, and eligible for a false GUITKX0105.
3. **Hook-prefix model mismatch** — compiler auto-prefixes 23 hook names only in setup/hook bodies (GX:373, 485, 632, 1164-1201); LSP stubs make bare hooks valid *everywhere* (VD:26-50, 442-448) — LSP-clean code fails at runtime.
4. **Typo recovery** — LSP recovers decl typos at edit-dist <=2 + length >=3 + shape gate (declScan.ts:43-78); compiler hint uses dist <=3, first identifier, no gate (GX:104-128); the LSP's own hint copies the compiler's threshold (declarations.ts:44), so the LSP disagrees even with itself.
5. **`@class_name` live check blind behind a leading comment** (declarations.ts:59 skips whitespace only) vs compiler comment-skipping (GX:39, 1228-1235).
6. **Unclosed `return (`** — three behaviors for one input: compiler GUITKX0304 error (GX:506-508), formatter verbatim fallback (FMT:695), virtualDoc empty-window-continue (VD:496).
7. **`@uss`** advertised in LSP completion (SCH:77), zero compiler handling.
8. **Markup parse errors discarded live** — `markup.ts` computes them, `scanWindowDiagnostics` throws them away (SRV:1494-1496) — filed **G5**.

Even where the parsers agree, the hand-parity has **no enforcement**: no contract test compares GD and TS outputs; the sidecar FNV-1a hash pair (GC:23-28 = diagsSidecar.ts:19-28) is byte-identical by construction, not by test.

### 5.2 How the Unity side avoids this

One parser, `ReactiveUITK.Language` (`U/ide-extensions~/language-lib/`), is **compiled into both worlds**: the Roslyn source generator and the LSP server reference the same assembly, and `HookRegistry.cs` is Compile-Include-linked from `Shared/Core/` so the generator, analyzer, LSP hover, and Unity runtime read one table. Where Unity *does* duplicate — the SG emitter vs the Editor HMR emitter — it guards with explicit contract tests (`Hmr*ContractTests`, `AsmdefResolverParityTests`). Where it duplicates *without* tests (DA counts directive roots for 0108; SV counts only elements — DA:207-253 vs SV:57-76), it exhibits exactly the Godot disease in miniature. Lesson: **share code where possible, contract-test where sharing is impossible.**

### 5.3 Recommendation

Make `guitkx.gd`'s parser family the canonical grammar and put the TS side under test against it:

1. **Golden-corpus contract tests** (first, cheapest): a directory of `.guitkx` fixtures; a Godot headless run dumps `{ast, diagnostics}` JSON per fixture via `RUIGuitkx`; a vitest suite runs `markup.ts`/`declScan.ts`/`virtualDoc.ts` over the same fixtures and diffs. Every item in 5.1 becomes a red test. Direct analogue of Unity's `Hmr*ContractTests`.
2. **Port `guitkx_jsx_scan.gd` to TS** with the same byte-identical discipline as `scanner.ts`, and have virtualDoc use it (closes 5.1 item 1 and most of G7/G8's blast radius).
3. **Single vocabulary module**: HOST_TAGS, hook-prefix list, directive keywords, and diagnostic message templates defined once (e.g. a JSON checked into `addons/reactive_ui/guitkx/` that both `guitkx.gd` and `schema.ts` load or are generated from) — the Godot equivalent of Unity's linked `HookRegistry.cs`.
4. Longer-term, the real fix is one implementation: since gdscript-analyzer (Rust) already ships as the LSP's analysis engine, the `.guitkx` markup grammar is a candidate for the same treatment (one Rust/napi core, thin GD binding for the compiler) — large; do 1-3 first.
