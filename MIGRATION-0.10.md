# Migrating to 0.10 — imports & exports

0.10.0 makes cross-file references **explicit and strict**: a `.guitkx` file that uses another
file's component, hook, or module must `import` it, and the target declaration must be marked
`export`. An implicit reference is now a compile error (`GUITKX2305`) whose message contains the
exact import line to add.

## TL;DR — one command

```bash
godot --headless --path . --script res://addons/reactive_ui/dev/migrate_0_10_0.gd
```

The codemod rewrites every `.guitkx` in your project in place:

1. Prefixes **`export `** onto every top-level declaration (export-everything — privacy is opt-in
   going forward: remove `export` from anything you want file-private).
2. Writes one **`import { … } from "…"` line per referenced file** — detected from your markup tags
   (`<StatusChip />`) and qualified references (`MyHooks.use_thing(...)`). Sibling files get a
   `./name` specifier; anything else gets a `~/` root-relative one.

It is **idempotent and re-runnable** — running it twice changes nothing, and it's safe to re-run
after a rebase or on a partially-migrated tree.

Then rebuild once (the watcher does this on the next editor focus, or run the compile sweep) and
you're done.

## What the codemod does NOT touch

- **Hand-written `class_name` scripts** (`DoomTypes`-style helpers, your game logic): these are
  *ambient* — referencing them never needs an import, and the codemod never rewrites `.gd` files.
- **Host/engine tags** (`<VBoxContainer>`, `<Label>`, …): engine vocabulary, never imported.
- **Generated `.gd` files**: regenerated by the compiler, not migrated.

## What changed underneath (nothing you need to act on)

- **A file may now hold several declarations** (components + hooks + modules). One-per-file remains
  the recommended convention; the binding (`class_name`) is the `@class_name` override, else the
  first exported declaration.
- **Component imports stay lazy** (`V.comp`) — cross-file component cycles remain legal.
  **Hook/module imports are eager preloads** — a cycle between them is now a clear error
  (`GUITKX2306`) that prints the chain.
- **`~/` specifiers** resolve against the `"root"` key of the nearest `guitkx.config.json`
  (default `res://`) — set it if your UI lives under a source root like `res://ui`.
- **`@uss` / `@theme` now also accept `~/` paths** (in addition to `res://` and `uid://`).
- New diagnostics **`GUITKX2300`–`GUITKX2309`** cover every import mistake — see the docs site's
  Diagnostics page.

## If something errors after migrating

| Error | Meaning | Fix |
|---|---|---|
| `GUITKX2305` | a cross-file reference the codemod's scan missed (e.g. a name built dynamically) | add the import line printed in the message |
| `GUITKX2301` | you removed `export` from something another file still imports | re-add `export`, or remove the stale import |
| `GUITKX2306` | two hook/module files import each other | break the chain, or make the cyclic edge a component reference (those may cycle) |
| `GUITKX2304` (warning) | an import the codemod added is no longer used | delete the line |

The IDE tooling (VS Code / VS 2022 `0.10.0`, editor addon `0.8.0`) treats imports as optional
syntax — an un-migrated project stays green in the editor until you run the codemod.
